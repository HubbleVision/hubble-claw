## 自定义开发

1. skill编排
    - 专门skill负责流程编排planner, 对接给各个流程的Skill
2. 记忆系统
    - 对话记忆：每轮次的对话都进行存储，对话发生的过程中有SessionMemory 做实时蒸馏。监测当前session的变化进行对话增量存储，从存储中提取值得持久化的内容，写入带 YAML 头的 MD 文件。多session的情况也进行相应处理。

        - 当用户上传文档或文章，一个Skill负责解析这类文档的结构。

        - 当提问具体业务问题时，负责回答问题的Skill会生成一条CLI的调用链路，分层召回内容（避免一次召回太多内容占用上下文），最后基于召回的数据回答。
3. 性能
    - 性能：如果是通过webhook或ws调用openclaw的实现方式，是否有用户并发数不超过3个的问题？

```
用户发送 skill 链接/名字
  → 解析 skill 元信息（名称、描述、来源、权限）
```

## 4. 记忆系统 `[自定义]`

自定义开发的**核心重点**，分两层。

### 4.1 对话记忆（Session Memory）

**流程**：

```
实时对话流
  → SessionMemory 实时蒸馏（每轮对话提取关键信息）
  → 监测 session 变化，增量存储对话
  → 跨 session 合并：多 session 的记忆去重 + 合并
  → 持久化：值得长期保留的内容写入带 YAML 头的 MD 文件
```

**存储格式**：

```yaml
---
type: conversation_memory
session_id: "s_20260402_001"
analyst: "宏观分析师"
user: "微信用户A"
created: 2026-04-02T14:30:00+08:00
tags: ["宏观", "美联储", "降息"]
tickers: ["600519.SH", "000858.SZ"]                 
industry_ids: ["SW-食品饮料", "SW-白酒"]                                     macro_topic_ids: ["FED-利率决议", "CN-消费"]  
market: "CN"          # 市场：CN / US / HK / CRYPTO
sentiment: "bullish"  # 观点倾向：bullish / bearish / neutral

---

- 用户关注美联储 2026 Q2 降息预期
- 用户偏好看多中国A股消费板块
- 上次讨论结论：建议关注食品饮料ETF
```

**需开发**：

- **SessionMemory 蒸馏**：实时提取对话关键信息
- **增量存储**：监测变化、写入存储
- **跨 session 合并**：去重、冲突解决、时效性判断

### 4.2 文档知识库

用户上传文档时触发三阶段流水线：

**阶段 1 — 文档解析 Skill**

- 输入：用户上传的文档/文章（PDF、公众号、网页）
- 输出：结构化文档树（标题层级、段落、表格、图表描述）

**阶段 2 — 分层检索库构建 Skill**

- 输入：结构化文档树
- 处理：生成 CLI 可调用的分层索引
    - L1：文档摘要（一句话概括）
    - L2：章节摘要（每章核心观点）
    - L3：段落级内容（原文 chunk）
- 输出：可供 CLI 调用的检索库

**阶段 3 — 问答召回 Skill**

- 输入：用户问题
- 处理：生成 CLI 调用链路，分层召回
    - 先召回 L1 判断相关文档
    - 再召回 L2 定位相关章节
    - 最后召回 L3 获取具体内容
- 输出：基于召回内容的回答（避免一次性召回过多占用上下文）

**需开发**：三个阶段全部需要开发。

### 4.3 性能注意事项

> **待验证**：通过 webhook/ws 调用 OpenClaw 的实现方式是否有并发数限制（如不超过 3 个）？
> 

路演 Demo 场景下需确保至少支持：4 个分析师实例 + 若干演示用户同时在线。建议在部署阶段进行并发压测确认。

## 5. 流程编排（Planner Skill） `[自定义]`

一个专门的 planner skill 负责协调各 skill 的执行顺序，是分析师的"调度中心"。

### 5.1 工作机制

```
用户输入
  → Planner 解析意图，判断需要哪些 skill、什么顺序
  → 生成执行计划（DAG）
  → 按依赖关系调度 skill 执行
      → 可并行的 skill 并行执行
      → 有依赖的 skill 串行执行
  → 汇总各 skill 输出，生成最终回答
```

### 5.2 示例

用户输入："帮我分析一下贵州茅台"

```yaml
plan:
  - step: 1
    parallel:
      - skill: hubble-stock-data
        args: { ticker: "600519.SH" }
      - skill: news-search
        args: { query: "贵州茅台 最新消息" }
  - step: 2
    skill: stock-analysis
    args: { data: "$step1.hubble-stock-data", news: "$step1.news-search" }
  - step: 3
    skill: generate-response
    args: { analysis: "$step2" }
```

## 6. 微信交互流程

### 6.1 对话体验规范 `[自定义]`

**即时反馈**：收到用户消息后，先立即回复"正在分析中..."（或带有分析师人设的思考提示，提供网页链接显示思考过程），分析完成后再发送完整回答。

**猜测追问**：每轮回答末尾附带 2-3 个猜测用户可能追问的选项，方便演示时快速点选：

```
📊 贵州茅台（600519）近期分析：
...（分析内容）...

---
您可能还想了解：
1️⃣ 茅台近半年资金流向如何？
2️⃣ 白酒板块整体趋势怎么样？
3️⃣ 有哪些相关的最新政策消息？
```

### 6.2 内容接收处理 `[自定义]`

| 用户发送内容 | 处理流程 |
| --- | --- |
| 股票 ticker（如"600519"） | planner 调度 stock-analysis + hubble 数据 → 输出分析报告 |
| 微信公众号文章链接 | 公众号浏览 skill 解析 → 语义记忆存储 → 可基于内容问答 |
| 网页链接 | 网页浏览 skill 解析 → 语义记忆存储 → 可基于内容问答 |
| 推文链接 | 推文浏览 skill 解析 → 语义记忆存储 → 可基于内容问答 |
| PDF 文件 | PDF 浏览 skill 解析 → 语义记忆存储 → 可基于内容问答 |
| skill 链接/名字 | 触发自主安装流程（见 3.2） |
| 文字问题 | planner 根据上下文调度对应 skill |

### 6.3 定时任务 `[自定义]`

用户可通过自然语言创建定时任务：

```
用户："每周一早上 9 点帮我总结上周 A 股大盘表现"
  → planner 识别为定时任务意图
  → 创建 cron 任务：0 9 * * 1
  → 绑定执行计划：news-search + hubble-stock-data + 生成周报
  → 回复用户确认
  → 到时间自动执行，结果推送到微信
```

**支持的定时任务类型**：

- 定期总结（日报/周报）
- 定期投研（跟踪特定 ticker）
- 自定义触发条件（如"茅台跌超 3% 时提醒我"— 依赖 stock-watcher）

**需开发**：定时任务调度 + 自然语言解析 cron + 结果推送。
